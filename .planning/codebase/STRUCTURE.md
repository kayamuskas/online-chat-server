## Directory Purposes

**`.planning/`:**
- Purpose: Hold planning and analysis outputs rather than product runtime code.
- Contains: Generated documents under `.planning/codebase/`.
- Key files: `.planning/codebase/ARCHITECTURE.md`, `.planning/codebase/STRUCTURE.md`

**`requirements/`:**
- Purpose: Store the authoritative product requirements and prototype artifacts.
- Contains: Markdown specifications plus the `desing_v1` prototype directory.
- Key files: `requirements/requirements_raw.md`, `requirements/wireframes.md`

**`requirements/desing_v1/`:**
- Purpose: Contain the first browser-viewable wireframe implementation.
- Contains: `index.html`, `styles.css`, a `components/` folder, and static upload references.
- Key files: `requirements/desing_v1/index.html`, `requirements/desing_v1/styles.css`

**`requirements/desing_v1/components/`:**
- Purpose: Separate the prototype into shared UI primitives and flow-specific JSX files.
- Contains: Plain `.jsx` scripts executed in browser order, not ES modules.
- Key files: `requirements/desing_v1/components/primitives.jsx`, `requirements/desing_v1/components/main_chat.jsx`, `requirements/desing_v1/components/tweaks.jsx`

**`requirements/desing_v1/uploads/`:**
- Purpose: Keep reference assets associated with the wireframe effort.
- Contains: A PDF requirements artifact.
- Key files: `requirements/desing_v1/uploads/2026_04_18_AI_herders_jam_-_requirements_v3.pdf`

## Naming Conventions

**Files:**
- Use lowercase names with underscores for top-level requirement documents: `requirements/requirements_raw.md`, `requirements/wireframes.md`.
- Use lowercase `.jsx` files by feature area inside `requirements/desing_v1/components/`: `main_chat.jsx`, `personal.jsx`, `manage.jsx`.
- Keep CSS and HTML filenames conventional and flat inside the prototype root: `requirements/desing_v1/index.html`, `requirements/desing_v1/styles.css`.

**Directories:**
- Use lowercase directory names.
- Keep prototype-specific files nested under `requirements/desing_v1/` rather than introducing an application `src/` tree before a real app exists.
- Preserve the current spelling of `requirements/desing_v1/` when adding adjacent files, because existing references in `requirements/desing_v1/index.html` depend on that exact path.

## Where to Add New Code

**New Requirement or Product Rule:**
- Primary code: `requirements/requirements_raw.md`
- Supporting wireframe note: `requirements/wireframes.md`

**New Prototype Screen or Flow:**
- Implementation: Add a new file under `requirements/desing_v1/components/` if it represents a distinct product area, or extend the closest existing flow file if it belongs to an established area.
- Composition: Register the new component with `Object.assign(window, ...)` and include its `<script type="text/babel" ...>` tag plus its rendered usage in `requirements/desing_v1/index.html`.

**New Shared UI Primitive:**
- Shared helpers: `requirements/desing_v1/components/primitives.jsx`
- Styling: `requirements/desing_v1/styles.css`

**New Static Reference Asset:**
- Supporting files: `requirements/desing_v1/uploads/`

**Production Application Code:**
- Not applicable in the current repository state. There is no established `src/`, backend package, or deployment structure to extend.
- If production code is introduced later, document the new root layer explicitly instead of mixing it into `requirements/desing_v1/`.

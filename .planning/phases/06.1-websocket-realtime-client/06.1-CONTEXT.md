# Phase 6.1: WebSocket Real-Time Client - Context

**Gathered:** 2026-04-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Wire the React frontend to the existing Socket.IO gateways so that newly created messages, edited messages, friend-request notification badge updates, and contact presence changes all arrive without a page refresh. This phase is pure client-side integration — no new backend gateway events are added. REST remains the authoritative path for all mutations and history recovery (D-33 from Phase 6).

</domain>

<decisions>
## Implementation Decisions

### Socket Client Architecture

- **D-01:** A single Socket.IO connection is maintained at the React application level via a context/provider. Architecture details (singleton module, React context shape, hook API) are **Claude's Discretion** — the user is not a frontend specialist and delegates the implementation approach entirely. The provider should connect after authentication succeeds and disconnect on sign-out.

### Friend Request Notifications

- **D-02:** Friend request badge updates use **polling only** — no new backend push events are added in Phase 6.1. The AppGateway is not extended for friend-request events in this phase.

### Presence Updates

- **D-03:** Presence status uses **on-demand polling** via the existing `getPresence` WebSocket event (request/response pattern already implemented in AppGateway). No server-push presence broadcasts are added. The sidebar or presence components call `getPresence` when they need fresh data.

### Message Real-Time Updates

- **D-04:** `RoomChatView` and `DmChatView` subscribe to `message-created` and `message-edited` events from `MessagesGateway`. On entering a conversation the client emits `joinRoom` or `joinDm`; on leaving it emits `leaveRoom` or `leaveDm`.

### Autoscroll Behavior

- **D-05:** Smart autoscroll — the timeline does **not** force-scroll to the bottom when a new message arrives if the user has scrolled up. When scrolled up and a new real-time message arrives, a **"↓ новые сообщения" button** appears (or equivalent badge/indicator). Clicking it scrolls to the bottom and dismisses the indicator.

### Reconnection Strategy

- **D-06:** On WebSocket reconnect, silently re-subscribe to the active conversation channel (`joinRoom`/`joinDm`) and **refetch history from the last known watermark** to catch any messages missed during the disconnect. No user-visible "reconnecting" spinner is required.

### Claude's Discretion

- Exact shape of the socket context (React Context, Zustand, or lightweight module singleton) — choose whichever fits existing frontend patterns.
- Whether the "↓ новые сообщения" button is a floating pill, a sticky bar, or a badge — any clear affordance is acceptable.
- Refetch debounce/throttle strategy on reconnect — avoid hammering the REST endpoint on flapping connections.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `MessagesGateway` (backend): fully implemented — joinRoom/joinDm/leaveRoom/leaveDm handlers and broadcastMessageCreated/broadcastMessageEdited exist. Frontend just needs to connect and listen.
- `AppGateway` (backend): getPresence is a working request/response WS event. No changes needed.
- `MessageTimeline`: exists with scroll container — autoscroll logic additions land here.
- `RoomChatView` / `DmChatView`: conversation lifecycle (mount/unmount) is the natural place for joinRoom/joinDm/leaveRoom/leaveDm calls.

### Established Patterns
- Auth pattern: session cookie passed in WS handshake (same as AppGateway — `extractSessionToken` from handshake cookie). Frontend must pass `withCredentials: true` to socket.io-client.
- Both gateways run on the **default namespace `/`** (no namespace param in decorator) — one socket connection serves all events.
- REST history uses `lastWatermark` / `firstWatermark` / `hasMoreBefore` fields — reconnect refetch uses `lastWatermark` as cursor.

### Integration Points
- `apps/web/src/App.tsx`: wrap authenticated shell with socket provider; connect on `handleAuthenticated`, disconnect on `handleSignedOut`.
- `apps/web/package.json`: `socket.io-client` is not yet installed — must be added as a dependency.

</code_context>

<specifics>
## Specific Ideas

- The "↓ новые сообщения" button should appear in the timeline area (floating above the composer), not in the topbar.
- Reconnect refetch: use `GET /api/v1/messages/room/:roomId/history?afterWatermark={lastWatermark}` (or equivalent DM endpoint) to fetch only missed messages — do not reload the full history page.

</specifics>

<deferred>
## Deferred Ideas

- Server-push friend request notifications (AppGateway broadcasting `friend-request-received`) — deferred, polling is sufficient for Phase 6.1.
- Server-push presence broadcasts (AppGateway broadcasting status changes to subscribers) — deferred, on-demand poll is sufficient.
- Typing indicators — out of scope for Phase 6.1.
- Read receipts — out of scope for Phase 6.1.

</deferred>

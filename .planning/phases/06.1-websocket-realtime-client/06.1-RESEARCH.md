## Summary

Phase 6.1 — чисто клиентская интеграция. Вся серверная часть (MessagesGateway с joinRoom/leaveRoom/joinDm/leaveDm, broadcastMessageCreated/broadcastMessageEdited, AppGateway с getPresence) уже реализована и работает. Задача фазы — добавить `socket.io-client` во frontend и подключить существующие компоненты к работающим backend events.

Ключевые технические находки:
1. Бэкенд использует `socket.io ^4.8.1` на порту 3000 (HTTP), с session-cookie auth (`withCredentials: true` обязателен).
2. Оба gateway (`AppGateway` и `MessagesGateway`) работают на **дефолтном namespace `/`** — один socket обслуживает все события.
3. Reconnect recovery через `after_watermark` **не реализован** в текущем API. Единственный параметр пагинации — `before_watermark` (страницы назад). Для reconnect нужна стратегия обходного пути или расширение API.
4. `RoomChatView` и `DmChatView` уже содержат плейсхолдер-комментарии «Phase 6 does not implement WS subscription here» — места вставки очевидны.
5. В frontend нет React Context, нет singleton socket модуля — создаётся с нуля.
6. `socket.io-client` не установлен в `apps/web/package.json`.

**Primary recommendation:** Создать `apps/web/src/lib/socket.ts` как singleton-модуль (не React Context) с функциями `connectSocket(url)` и `getSocket()`, затем обернуть App.tsx в тонкий `SocketProvider` только для propagation, чтобы дочерние компоненты получали socket через хук `useSocket()`.

### Pattern 1: Singleton Socket Module

**What:** Модуль-уровневый singleton, чтобы избежать множественных соединений при ре-рендерах React.

**When to use:** Всегда — один socket на всё приложение.

```typescript
// Source: https://socket.io/docs/v4/client-api
// apps/web/src/lib/socket.ts

import { io, Socket } from "socket.io-client";
import { SERVICE_PORTS } from "@chat/shared";

let socket: Socket | null = null;

export function connectSocket(): Socket {
  if (socket?.connected) return socket;
  socket = io(`http://localhost:${SERVICE_PORTS.apiHttp}`, {
    withCredentials: true,   // session cookie required by ws-auth.ts
    autoConnect: true,
  });
  return socket;
}

export function getSocket(): Socket | null {
  return socket;
}

export function disconnectSocket(): void {
  socket?.disconnect();
  socket = null;
}
```

### Pattern 2: Thin SocketProvider + useSocket Hook

**What:** React Context передаёт socket ссылку компонентам; connect/disconnect привязан к auth-статусу.

**When to use:** В App.tsx и всех дочерних компонентах, которым нужен socket.

```typescript
// apps/web/src/features/socket/SocketProvider.tsx (or inline in App.tsx)
import { createContext, useContext, useEffect, useRef } from "react";
import { connectSocket, disconnectSocket, getSocket } from "../../lib/socket";
import type { Socket } from "socket.io-client";

const SocketContext = createContext<Socket | null>(null);

export function SocketProvider({ authenticated, children }: {
  authenticated: boolean;
  children: React.ReactNode;
}) {
  const socketRef = useRef<Socket | null>(null);

  useEffect(() => {
    if (authenticated) {
      socketRef.current = connectSocket();
    } else {
      disconnectSocket();
      socketRef.current = null;
    }
  }, [authenticated]);

  return (
    <SocketContext.Provider value={socketRef.current}>
      {children}
    </SocketContext.Provider>
  );
}

export function useSocket(): Socket | null {
  return useContext(SocketContext);
}
```

### Pattern 3: Conversation Subscription in Chat Views

**What:** useEffect lifecycle для joinRoom/leaveRoom с reconnect recovery.

**When to use:** В RoomChatView и DmChatView на mount/unmount.

```typescript
// Source: MessagesGateway events (verified: apps/api/src/messages/messages.gateway.ts)
// Pattern for RoomChatView (DmChatView mirrors with joinDm/leaveDm/conversationId)

useEffect(() => {
  if (!socket || !roomId) return;

  socket.emit("joinRoom", { roomId });

  function onMessageCreated(payload: MessageCreatedPayload) {
    // append to messages list; trigger autoscroll logic
    setMessages(prev => {
      // dedup guard (reconnect may replay)
      if (prev.some(m => m.id === payload.message.id)) return prev;
      return [...prev, mapPayloadToView(payload.message)];
    });
    setHasNewMessages(true);
  }

  function onMessageEdited(payload: MessageEditedPayload) {
    setMessages(prev =>
      prev.map(m => m.id === payload.message.id ? { ...m, ...payload.message } : m)
    );
  }

  // Reconnect: re-subscribe and refetch missed messages
  function onReconnect() {
    socket.emit("joinRoom", { roomId });
    // Debounced refetch — see Pattern 4
    scheduleReconnectRefetch(roomId, lastWatermarkRef.current);
  }

  socket.on("message-created", onMessageCreated);
  socket.on("message-edited", onMessageEdited);
  socket.io.on("reconnect", onReconnect);

  return () => {
    socket.emit("leaveRoom", { roomId });
    socket.off("message-created", onMessageCreated);
    socket.off("message-edited", onMessageEdited);
    socket.io.off("reconnect", onReconnect);
  };
}, [socket, roomId]);
```

### Pattern 4: Reconnect Recovery (Workaround for Missing after_watermark)

**What:** Бэкенд не поддерживает `after_watermark`; для reconnect нужно загрузить последнюю страницу истории и смёрджить с существующим списком по ID.

**When to use:** В обработчике `reconnect` в chat views.

```typescript
// Reconnect recovery strategy (no after_watermark on backend)
const reconnectTimerRef = useRef<ReturnType<typeof setTimeout> | null>(null);

function scheduleReconnectRefetch() {
  if (reconnectTimerRef.current) clearTimeout(reconnectTimerRef.current);
  reconnectTimerRef.current = setTimeout(async () => {
    try {
      const result = await getRoomHistory(roomId); // last page, no cursor
      setMessages(prev => {
        const existingIds = new Set(prev.map(m => m.id));
        const newMsgs = result.messages.filter(m => !existingIds.has(m.id));
        return newMsgs.length > 0 ? [...prev, ...newMsgs].sort(
          (a, b) => a.conversationWatermark - b.conversationWatermark
        ) : prev;
      });
    } catch { /* silent — user will see stale history */ }
  }, 500); // D-06: debounce 500ms
}
```

### Pattern 5: Smart Autoscroll in MessageTimeline

**What:** Scroll listener + isScrolledUp state + "↓ новые сообщения" кнопка.

**When to use:** В MessageTimeline, получает `hasNewMessages` prop от родителя.

```typescript
// Source: D-05 spec + UI-SPEC.md threshold = 100px
const scrollRef = useRef<HTMLDivElement>(null);
const [isScrolledUp, setIsScrolledUp] = useState(false);

function handleScroll() {
  const el = scrollRef.current;
  if (!el) return;
  const distFromBottom = el.scrollHeight - el.scrollTop - el.clientHeight;
  setIsScrolledUp(distFromBottom > 100);
}

// Auto-scroll if not scrolled up
useEffect(() => {
  if (!isScrolledUp && messages.length > 0) {
    scrollRef.current?.scrollTo({ top: scrollRef.current.scrollHeight });
  }
}, [messages.length, isScrolledUp]);

// "↓ новые сообщения" button: show when isScrolledUp && hasNewMessages
// Click: scrollRef.current?.scrollTo({ top: scrollRef.current.scrollHeight, behavior: "smooth" })
```

### Pattern 6: Presence Polling in ContactsSidebar

**What:** getPresence WS request/response, polling каждые 30с.

**When to use:** В ContactsSidebar при наличии socket и списка contacts.

```typescript
// Source: AppGateway handleGetPresence (verified: apps/api/src/ws/app.gateway.ts)
// Server emits "presence" PresenceMap in response

useEffect(() => {
  if (!socket || contacts.length === 0) return;

  const userIds = contacts.map(c => c.userId);

  function poll() {
    socket!.emit("getPresence", { userIds });
  }

  function onPresence(presenceMap: Record<string, { status: string }>) {
    // update each contact's presenceStatus in parent state
    onPresenceUpdate(presenceMap);
  }

  socket.on("presence", onPresence);
  poll(); // immediate on mount
  const interval = setInterval(poll, 30_000); // D-03: 30s interval

  return () => {
    socket.off("presence", onPresence);
    clearInterval(interval);
  };
}, [socket, contacts.map(c => c.userId).join(",")]);
```

### Anti-Patterns to Avoid

- **Multiple socket connections:** Не вызывать `io()` внутри компонентов — всегда через singleton `getSocket()`. React renders = множественные соединения.
- **Missing withCredentials:** Без этого флага session cookie не отправляется, сервер отключает socket сразу (ws-auth.ts).
- **socket.on без cleanup:** Всегда снимать listeners в return() useEffect. Иначе дублирование обработчиков при ремонте компонентов.
- **socket.io.on vs socket.on для reconnect:** Reconnect events (`reconnect`, `reconnect_attempt`) принадлежат Manager (`socket.io`), а НЕ Socket (`socket`). Использовать `socket.io.on("reconnect", ...)`.
- **leaveRoom при размонтировании:** Не эмитить leaveRoom при переключении вкладок App — только при размонтировании самого RoomChatView.
- **Full history reload вместо merge:** Reconnect не должен заменять весь messages-список, только добавлять пропущенные. Иначе теряется scroll-позиция.
- **Context value обновление при каждом render:** SocketProvider не должен использовать `useState` для socket — это вызовет ре-рендер всего дерева. Использовать `useRef`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| WebSocket transport с автореконнектом | Свой EventSource/WebSocket | socket.io-client | Автореконнект, back-off, polling fallback, namespace routing — всё встроено |
| Cookie pass для WS handshake | Ручное добавление заголовков | `withCredentials: true` в `io()` | Браузер автоматически добавляет cookies при withCredentials |
| Reconnect event system | `setInterval` опрос состояния | `socket.io.on("reconnect", ...)` | Manager events встроены в socket.io-client |
| Dedup при reconnect | Timestamp-сравнение | ID-based dedup (`Set<string>`) | Watermarks монотонные и уникальные по conversationId — ID надёжнее |

**Key insight:** socket.io-client решает весь transport layer (reconnect, buffering, polling fallback). Клиентский код должен только управлять subscription lifecycle и state merging.

## Common Pitfalls

### Pitfall 1: CORS mismatch при withCredentials

**What goes wrong:** Socket connect завершается ошибкой `connect_error` из-за CORS отказа.

**Why it happens:** Backend `MessagesGateway` объявлен с `cors: { origin: '*' }`. При `withCredentials: true` браузер требует конкретного origin, а не `*`.

**How to avoid:** Проверить backend CORS config перед тестированием. Если нужно — поменять на `origin: 'http://localhost:4173'` (WEB_PORT).

**Warning signs:** `connect_error` в console сразу при подключении; socket.active === false.

**CRITICAL:** [VERIFIED: apps/api/src/messages/messages.gateway.ts line 78] `@WebSocketGateway({ cors: { origin: '*' } })` — нужно будет изменить на конкретный origin при `withCredentials: true`, иначе браузер заблокирует соединение.

### Pitfall 2: socket.on("reconnect") вместо socket.io.on("reconnect")

**What goes wrong:** Reconnect handler никогда не вызывается.

**Why it happens:** В socket.io v3+ reconnect events переехали с Socket на Manager (`socket.io`). [CITED: https://socket.io/docs/v4/client-socket-instance]

**How to avoid:** Всегда использовать `socket.io.on("reconnect", handler)` для reconnect, `socket.io.off(...)` для cleanup.

**Warning signs:** Reconnect recovery не работает после network drop.

### Pitfall 3: Дублирование event handlers при ремонте компонента

**What goes wrong:** При каждом render RoomChatView добавляется новый listener; сообщения добавляются N раз.

**Why it happens:** `useEffect` без cleanup, или dependency array неверный (вызывает переподключение при каждом render).

**How to avoid:** Всегда возвращать cleanup function с `socket.off(event, handler)`. Именованные функции-handlers (не анонимные) обязательны для `.off()`.

**Warning signs:** Сообщения дублируются в timeline после навигации.

### Pitfall 4: Reconnect recovery заменяет весь messages state

**What goes wrong:** Пользователь теряет позицию прокрутки; история сбрасывается при каждом reconnect.

**Why it happens:** `setMessages(result.messages)` вместо merge.

**How to avoid:** ID-based merge: `result.messages.filter(m => !existingIds.has(m.id))`. Append только новые.

### Pitfall 5: after_watermark не существует на бэкенде

**What goes wrong:** REST-запрос `GET /history?afterWatermark=X` возвращает полную последнюю страницу (параметр игнорируется).

**Why it happens:** Текущий MessagesController поддерживает только `before_watermark`. `after_watermark` не реализован. [VERIFIED: apps/api/src/messages/messages.controller.ts]

**How to avoid:** Reconnect recovery должен загружать последнюю страницу (без параметров) и делать merge по ID, а не полагаться на `after_watermark`. Если нужна точная recovery — добавить `after_watermark` в API (задача для Wave 0 или отдельный план).

**Warning signs:** CONTEXT.md упоминает `afterWatermark` как будущий параметр, но он не существует сейчас.

### Pitfall 6: SocketProvider re-renders дерево при каждом изменении socket state

**What goes wrong:** При connect/disconnect всё дерево перерисовывается, включая message history.

**Why it happens:** Хранение socket в `useState` вместо `useRef`.

**How to avoid:** Socket ref в `useRef`; контекст передаёт ref.current (или сам `RefObject`). Компоненты получают socket и подписываются напрямую.

## Critical Finding: CORS Must Be Fixed

[VERIFIED: apps/api/src/messages/messages.gateway.ts]

`MessagesGateway` объявлен как:
```typescript
@WebSocketGateway({ cors: { origin: '*' } })
```

При использовании `withCredentials: true` (обязательно для session-cookie auth), браузер **блокирует** соединение, если сервер возвращает `Access-Control-Allow-Origin: *`. Нужно заменить на конкретный origin.

Аналогично проверить `AppGateway`:
[VERIFIED: apps/api/src/ws/app.gateway.ts line 38] `@WebSocketGateway({ cors: { origin: '*' } })` — та же проблема.

**Оба gateway нужно исправить** до первого тестирования WebSocket из браузера. Исправление: `cors: { origin: 'http://localhost:4173', credentials: true }`.

## Security Domain

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | yes | Session cookie передаётся через `withCredentials: true`; ws-auth.ts на бэкенде проверяет токен |
| V3 Session Management | yes | Socket disconnect на sign-out; socket не сохраняет credentials |
| V4 Access Control | partial | joinRoom/joinDm не проверяют membership на gateway (T-06-06 в коде); enforcement на HTTP layer |
| V5 Input Validation | no | Фаза только читает события; не отправляет user-authored data через WS |
| V6 Cryptography | no | TLS — ответственность deployment, не application code |

### Known Threat Patterns for Socket.IO Client

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Подключение без аутентификации | Spoofing | withCredentials: true + сервер отключает неаутентифицированные сокеты |
| XSS → кража session cookie | Tampering | HttpOnly cookie (уже выставлен auth.controller.ts); не меняется в этой фазе |
| Event listener leak → memory | Denial | Cleanup в useEffect return; named handler references |
| Reconnect storm при flapping | Denial | Debounce 500ms (D-06); socket.io built-in back-off |
| CORS wildcard + withCredentials | Tampering | Исправить gateway CORS с `*` на конкретный origin (Wave 0 gap) |

## Code Examples

### Инициализация socket с session-cookie auth

```typescript
// Source: https://socket.io/docs/v4/client-options (withCredentials)
// Source: ws-auth.ts pattern (cookie name "session")
import { io } from "socket.io-client";
import { SERVICE_PORTS } from "@chat/shared";

const socket = io(`http://localhost:${SERVICE_PORTS.apiHttp}`, {
  withCredentials: true,  // sends "session" cookie on WS handshake
});
```

### Подписка на reconnect через Manager

```typescript
// Source: https://socket.io/docs/v4/client-socket-instance
// reconnect is a Manager event, NOT a Socket event (changed in v3)
socket.io.on("reconnect", (attempt) => {
  // re-join active conversation
  socket.emit("joinRoom", { roomId });
  // schedule REST refetch with 500ms debounce
});
```

### Отписка именованного handler

```typescript
// Correct: named function allows .off() to remove specific handler
function onMessageCreated(payload: unknown) { /* ... */ }
socket.on("message-created", onMessageCreated);
// cleanup:
socket.off("message-created", onMessageCreated);

// Wrong: anonymous function cannot be removed
socket.on("message-created", (payload) => { /* ... */ });
// socket.off("message-created") removes ALL handlers — dangerous
```

### getPresence request/response

```typescript
// Source: AppGateway handleGetPresence (verified codebase)
// Client emits "getPresence", server responds with "presence" event
socket.emit("getPresence", { userIds: ["id1", "id2"] });
socket.on("presence", (presenceMap: Record<string, { status: string }>) => {
  // { "id1": { status: "online" }, "id2": { status: "offline" } }
});
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| socket.on("reconnect") | socket.io.on("reconnect") | socket.io v3 | Если использовать старый API — handler не вызывается |
| `polling` transport fallback | WebSocket first, polling fallback | socket.io v3+ | По умолчанию пробует WebSocket; не нужно настраивать |
| Separate namespaces per feature | Default namespace `/` для всех | Текущий проект | Один socket, один connection |

- socket.io v2 API: не применимо, проект на v4.

## Open Questions

1. **CORS fix — где именно поставить origin?**
   - What we know: Оба gateway объявлены с `cors: { origin: '*' }`. WEB_PORT = 4173 (из config.ts).
   - What's unclear: Dev vs prod — нужен ли переменный CORS origin из env?
   - Recommendation: В Wave 0 прописать `origin: 'http://localhost:4173'` хардкодом (dev-only); prod CORS — задача Phase 10/OPS.

2. **after_watermark для точного reconnect recovery**
   - What we know: Текущий API поддерживает только `before_watermark`. Reconnect recovery через merge последней страницы по ID работает, но может пропустить сообщения если за время disconnect пришло больше, чем PAGE_SIZE.
   - What's unclear: PAGE_SIZE — нужно проверить в messages.service.ts.
   - Recommendation: Плановый вариант — merge последней страницы (достаточно для Phase 6.1). Если нужна точность — Wave 0 добавляет `after_watermark` в API, но это выходит за boundary Phase 6.1 (pure client phase).

3. **SocketProvider placement в App.tsx**
   - What we know: App.tsx — крупный компонент, уже оборачивает все tabs.
   - What's unclear: Использовать отдельный файл `SocketProvider.tsx` или инлайн в App.tsx?
   - Recommendation: Отдельный файл `apps/web/src/features/socket/SocketProvider.tsx` — чище, проще тестировать изолированно.

<interfaces>
Presence semantics must preserve:

- `online` if at least one tab is active
- `AFK` only when all tabs are inactive for more than one minute
- `offline` only when all tabs are closed or offloaded
- durable `last seen`
- low-latency propagation compatible with the `<2s` source requirement

Do not fall back to frequent PostgreSQL polling for live presence.
</interfaces>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser activity -> websocket events | Presence truth depends on untrusted client-side activity signals |
| session cookie -> websocket identity | Realtime transport must not bypass existing auth/session boundaries |
| runtime presence -> durable `last seen` | Transition logic must avoid stale writes or DB-thrash patterns |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|

| T-03-06 | T/D | multi-tab presence aggregation | mitigate | Track per-tab runtime state with expiry/heartbeat semantics rather than trusting one socket or one disconnect event. |
| T-03-07 | D/I | durable `last seen` persistence | mitigate | Persist only on offline transition/write-behind boundaries so PostgreSQL does not become the live presence source. |

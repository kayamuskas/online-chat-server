## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline badge padding |
| sm | 8px | Compact element spacing, action button gaps |
| md | 16px | Default element spacing, card padding |
| lg | 24px | Section padding, modal body padding |
| xl | 32px | Layout gaps between sections |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

Exceptions:
- Message action bar buttons: 4px gap (`.msg-bubble__actions` gap: 0.4rem ≈ 6px — preserve existing value, do not change)
- Danger zone confirm inline block padding: 12px top/bottom, 16px left/right (matches `.revoke-confirm` pattern)
- `.btn--xs` padding: 0.25rem 0.625rem — preserve existing codebase value, do not change

Source: `apps/web/src/styles.css` — existing spacing pattern, 8-point scale already in use.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#fafaf7` (`--color-bg`) | Page background, message timeline background |
| Secondary (30%) | `#ffffff` (`--color-surface`) | Cards, panels, nav, modal background |
| Accent (10%) | `#b05a2c` (`--color-accent`) | Reply bar stripe, active nav items, focus rings, unread badges |

- Focus ring on inputs (`border-color: var(--color-accent)`)

Additional semantic tokens already in CSS:
- `--color-ok: #3a7d3a` — success confirmations (password verified, deletion initiated)

### 3. Room Delete Flow — inside `ManageRoomView` Settings tab

```
Location: apps/web/src/features/rooms/ManageRoomView.tsx — Settings tab
Pattern: two-step inline confirm (matches RevokeSessionConfirm from Phase 3)
Container: .danger-zone block appended after existing .manage-room__danger-card
```

States:
1. **Idle**: `<button class="btn btn--danger btn--xs">Delete Room</button>` — visible only to owner
2. **Confirming**: `.danger-zone__confirm` block expands inline
   - Text: "This will permanently delete **{room.name}**, all its messages and attachments. This cannot be undone."
   - Buttons: `<button class="btn btn--danger">Confirm Delete</button>` + `<button class="btn btn--soft">Keep Room</button>`
3. **Deleting**: both buttons disabled, Confirm shows "Deleting…"
4. **Done**: client navigates away on receiving `room-deleted` WS event — no success state needed

Confirm pattern: click-confirm (two-step button) — no type-name input. Source: CONTEXT.md Claude's Discretion, confirmed via `RevokeSessionConfirm.tsx` existing pattern.

### 4. Account Delete Flow — inside `AccountOverviewView`

```
Location: apps/web/src/features/account/AccountOverviewView.tsx
Container: .danger-zone block at bottom of account overview
```

States:
1. **Idle**: `<button class="btn btn--danger btn--xs">Delete Account</button>`
2. **Confirming**: `.danger-zone__confirm` block expands inline with password field
   - Password field: `.field__input` type="password" placeholder="Enter your password to confirm"
   - Buttons: `<button class="btn btn--danger" disabled={!password.trim()}>Confirm Delete Account</button>` + `<button class="btn btn--soft">Keep Account</button>`
3. **Submitting**: buttons disabled, Confirm shows "Deleting…"
4. **Error** (wrong password): `.error-msg` shown inline — "Incorrect password. Please try again."
5. **Success**: client redirected to auth screen (same as sign-out per D-16)

Password input: uncontrolled or controlled, cleared on cancel.

## Interaction Contracts

### MSG-05: Message Delete Button

```
Trigger:     Hover over .msg-bubble row
Visibility:  .msg-bubble__action--delete appears when:
               - conversation_type === 'room' AND (currentUserId === message.author_id OR isAdminOrOwner)
               - conversation_type === 'dm'  AND currentUserId === message.author_id
Animation:   Same fade-in as existing Reply/Edit buttons (CSS transition opacity 0.15s)
Click:       1. Call DELETE /api/v1/rooms/:roomId/messages/:messageId
                or DELETE /api/v1/dm/:conversationId/messages/:messageId
             2. Optimistic: remove message from local state immediately
             3. WS event 'message-deleted' confirms removal for other clients
             4. On API error: restore message + show inline error toast (not implemented in phase 8 — silent failure acceptable)
```

### ROOM-09: Room Delete Flow

```
Location:  ManageRoomView Settings tab (only rendered when currentUserId === room.owner_id)
Step 1:    User clicks "Delete Room" button
Step 2:    .danger-zone__confirm block expands (no modal, inline only)
           Text shows room name in bold
Step 3:    User clicks "Confirm Delete"
           - Button shows "Deleting…", both buttons disabled
           - Call DELETE /api/v1/rooms/:roomId
Step 4:    Server emits 'room-deleted' WS event (before data deletion per D-06)
Step 5:    Client receives 'room-deleted' → navigate to rooms list (no alert)
On error:  "Deletion failed. Please try again." inline in .danger-zone__confirm
           Buttons re-enabled
On cancel: collapse .danger-zone__confirm, return to Step 1
```

### AUTH-08: Account Delete Flow

```
Location:  AccountOverviewView danger zone (visible to all authenticated users)
Step 1:    User clicks "Delete Account" button
Step 2:    .danger-zone__confirm block expands with password input
Step 3:    User enters password + clicks "Confirm Delete Account"
           - Button shows "Deleting…", both buttons disabled, input disabled
           - Call DELETE /api/v1/auth/account with body { password }
Step 4:    On 200 OK: clear session cookie, navigate to /auth (same as sign-out)
On error (401/403 wrong password): show "Incorrect password. Please try again."
           Re-enable form, clear password field
On error (5xx): show "Deletion failed. Please try again."
           Re-enable form, preserve password field
On cancel: collapse confirm block, clear password field
```

### ROOM-07: Permission Matrix — No UI Change

The admin-cannot-ban-admin rule (D-17) is enforced server-side only. The ban button in `ManageRoomView` is already visible for admins against non-admin members. No UI change needed. The server returns `403 Forbidden` with message "Only the room owner can ban an admin." — the existing `actionError` state in `ManageRoomView` will surface this.

## Accessibility Requirements

- Delete buttons must have explicit `type="button"` (prevent form submission)
- Confirm/Cancel buttons must have explicit `type="button"`
- Password input for account deletion must have `autocomplete="current-password"`
- Danger zone expand/collapse must be immediate (no animation) — reduces vestibular motion issues
- Error messages must be rendered in the DOM (not tooltip-only) for screen reader access
- Delete button in message action bar: minimum 32px touch target height (`.btn--xs` achieves this)
- Destructive confirm button must remain disabled until password field has non-empty value (AUTH-08)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Docker daemon -> service containers | Compose orchestrates startup and readiness for dependent services |
| app containers -> Postgres/Redis | Backend and worker depend on healthy storage and queue services |
| container filesystems -> host volumes | Writable paths can expand attack and mutation surface if left open unnecessarily |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-10 | D | `infra/compose/compose.yaml` | mitigate | Use healthchecks plus `depends_on.condition=service_healthy` so startup race due to unhealthy service dependencies is blocked. |
| T-01-11 | I/D | Dockerfiles and Compose build policy | mitigate | Install from the committed lockfile during Docker build and forbid runtime dependency installs to block hidden startup/setup drift. |
| T-01-12 | T/E | container filesystem mounts | mitigate | Make service filesystems read-only by default and allow only narrowly scoped writable mounts, addressing unnecessarily writable container filesystems. |
| T-01-14 | T/I | lockfile-backed dependency delivery path | mitigate | Require the dependency smoke and documentation path to verify that Dockerfiles consume the committed lockfile workflow instead of silently drifting to unverified dependency inputs. |
</threat_model>

# Phase 9.1: Design Polish - Context

**Gathered:** 2026-04-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix design issues and visual inconsistencies between the current frontend and the `requirements/desing_v1/` design reference. This phase covers: registration inline validation, frozen DM redesign, ghost contact handling after account deletion, attachment card rendering, friend request UI (modal + member popover), and a design requirements audit against the Appendix checklist.

Phase 9.1 does NOT add new backend capabilities beyond what's needed to support the design fixes (e.g., username reservation table, account-deletion cascade changes). No new features — only alignment with the existing design spec.

</domain>

<decisions>
## Implementation Decisions

### 1. Register inline validation

- **D-9.1-01:** Inline validation (email/username availability, password strength) runs only on form submit, not on every keystroke. No debounce or realtime checking.
- **D-9.1-02:** After submit, display AVAILABLE/TAKEN badges inline next to email and username fields per design Variation E (`auth.jsx:AuthVariantE`).
- **D-9.1-03:** Password strength indicator ("strong"/"weak") shown after submit validation.
- **D-9.1-04:** Backend already has `findByEmail`/`findByUsername` in `UserRepository` — the registration endpoint already checks uniqueness. Frontend needs to parse the error response and show field-specific inline errors instead of a single generic error banner.
- **D-9.1-05:** Add "✎ username is immutable after sign-up" footer text below the Create account button per design.

### 2. Frozen DM — full header redesign

- **D-9.1-06:** Replace the current minimal `read-only` badge with a full "CONVERSATION FROZEN" banner block at the bottom of the message timeline, styled per `personal.jsx` banned variant: dashed border, warm background, monospace label "⊘ CONVERSATION FROZEN", ban date, explanation text, and "Unban to resume" link.
- **D-9.1-07:** DM header gets "View profile" and "Unban user" buttons per design when conversation is frozen.
- **D-9.1-08:** Composer area shows disabled state with centered monospace text "Messaging disabled — user is banned" instead of the regular composer.
- **D-9.1-09:** When not banned, DM header shows "View profile" and "Ban user" buttons per design.

### 3. Ghost contact (deleted user)

- **D-9.1-10:** When an account is deleted, freeze all DM conversations for the other party (set `frozen=true`) instead of deleting them. DM messages are already preserved via `SET NULL` on `author_id`.
- **D-9.1-11:** Show a "ACCOUNT DELETED" banner (not "CONVERSATION FROZEN") in the frozen DM, with text like "This user deleted their account on [date]. History is read-only." No unban/unblock buttons.
- **D-9.1-12:** In the contacts sidebar, show the deleted user's contact entry as "[deleted]" instead of the original username, with offline presence.
- **D-9.1-13:** Block username re-use: add a `deleted_usernames` table (or similar) populated on account deletion. Registration must check this table and reject previously-used usernames.
- **D-9.1-14:** Backend change: `deleteAccount` in `auth.service.ts` must freeze DM conversations (set `frozen=true`, record deletion timestamp) before deleting friendship rows, rather than deleting both friendships and DM conversations outright.
- **D-9.1-15:** Friendship rows should still be cleaned up on account deletion (current behavior). The frozen DM conversation + "[deleted]" display handle the UX. No ghost friendship entries.

### 4. Attachment card rendering + timeline dividers

- **D-9.1-16:** Replace the current plain link attachment rendering in `MessageTimeline` with a card component matching the design: file type icon (DOC/IMG/etc.), original filename, file size, optional comment, and a Download button.
- **D-9.1-17:** Verify "older messages ↑" divider matches design styling (dashed line + centered text). Currently shows as a button "Scroll up for earlier messages" — restyle to match design's `chat-divider` pattern.
- **D-9.1-18:** Verify "new ↓" divider styling matches design. Currently rendered in `msg-timeline__divider` — compare with design's `chat-divider` element.

### 5. Friend request UI

- **D-9.1-19:** Restyle `AddContactModal` to match design Variant B: title "ADD FRIEND BY USERNAME" in monospace, field labels "Username" (with @ prefix) and "Message (optional)", accent-colored "Send request" button, and footer "Requires confirmation from [name]."
- **D-9.1-20:** New component `MemberPopover` — shown on click of a room member row in the right panel. Displays: avatar circle with initial, name, @handle + presence dot, context line ("Member of #engineering. Not in your contacts."), and action menu: Send friend request / View profile / Ban user (in red). Styled per `contacts.jsx:MemberPopover`.
- **D-9.1-21:** MemberPopover replaces the current inline "Add friend" button in `RoomMembersTable`.

### 6. Design requirements audit

- **D-9.1-22:** During research/planning, systematically walk through the Appendix checklist in `index.html` lines 118-134 and compare each flow reference against current implementation. Any gaps beyond the 5 items above get added as additional plan tasks.

### Claude's Discretion

- Exact file type icon mapping for attachment cards (DOC, IMG, PDF, etc.) — Claude can decide the icon set and extension-to-icon mapping.
- CSS class naming for new components (MemberPopover, attachment card) — follow existing conventions in `styles.css`.
- Whether the "ACCOUNT DELETED" frozen DM uses the same component as the "CONVERSATION FROZEN" banned DM with a variant prop, or a separate conditional render.

</decisions>

<specifics>
## Specific Ideas

- The "CONVERSATION FROZEN" banner and "ACCOUNT DELETED" banner can share a base component with different props for icon, title, body text, and optional action link.
- MemberPopover should be positioned relative to the clicked member row and dismiss on outside click — standard popover pattern.
- Attachment card icons can use simple text labels (DOC, IMG, PDF, ZIP) in a styled box, matching the design's monospace icon approach — no need for an icon library.
- The "older messages ↑" divider should be a clickable affordance that triggers `onLoadOlder`, not just a visual separator.

</specifics>

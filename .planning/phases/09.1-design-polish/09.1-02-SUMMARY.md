# Phase 9.1 Plan 02: Registration Inline Validation Badges Summary

Registration form enhanced with field-level AVAILABLE/TAKEN badges, password strength indicator, and immutable username footer text per design AuthVariantE — all triggered on form submit only, no keystroke validation.

## What Was Built

### RegisterView.tsx changes
- Added three new state variables: `emailStatus`, `usernameStatus`, `passwordStrength` (all null initially, set only in `handleSubmit`)
- Added `checkPasswordStrength(pw)` helper: strong if 12+ chars, or 8+ with upper+lower+digit mix (weak otherwise)
- `handleSubmit` clears all status at start, computes password strength before API call
- Catch block parses `"email is already registered"` → sets `emailStatus("taken")`, leaves username null (never checked by backend); parses `"username is already taken"` → sets `usernameStatus("taken")`, `emailStatus("available")` (email passed)
- On success: sets both to "available" before calling `onSuccess`
- Badge `<span>` rendered below each input with `.badge.ok` (available/strong) or `.badge.warn` (taken/weak)
- Immutable username footer note added below the Create account button

### styles.css changes
- Added `.badge`, `.badge.ok`, `.badge.warn` CSS classes using design token fallbacks (`var(--color-ok)`, `var(--presence-afk)`)

## Deviations from Plan

### CSS Variable Naming Adjustment (Rule 1 - Bug Prevention)
- **Found during:** Task 1 implementation
- **Issue:** Design spec references `var(--ok)` and `var(--afk)`, but this project's CSS custom properties use `var(--color-ok)` and `var(--presence-afk)`. Using the design tokens verbatim would produce unstyled badges.
- **Fix:** Used project-native variables with hard-coded fallbacks: `color: var(--color-ok, #5a9a5a)` and `color: var(--presence-afk, #c9a227)`
- **Files modified:** apps/web/src/styles.css
- **Commit:** 7b81dcd

### Backend Pitfall 1 Alignment (Rule 2 - Correctness)
- **Found during:** Task 1 — plan draft showed `setUsernameStatus("available")` when email error fires
- **Issue:** RESEARCH.md Pitfall 1 explicitly warns: if email fails, username was never checked by backend, so showing "available" for username is misleading. The plan action already corrected this in step 5, but it was re-verified during implementation.
- **Fix:** When email error fires, `usernameStatus` is set to `null` (no badge), not "available". "Available" for email is only set when the username error fires (meaning email passed).
- **Commit:** 7b81dcd

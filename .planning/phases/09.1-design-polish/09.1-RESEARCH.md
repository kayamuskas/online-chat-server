## Summary

Phase 9.1 is a design polish pass that aligns the existing frontend with the `requirements/desing_v1/` design reference. The work spans 6 areas: (1) registration inline validation feedback, (2) frozen DM header/banner/composer redesign, (3) ghost contact handling after account deletion (including backend cascade changes and a new `deleted_usernames` table), (4) attachment card rendering and timeline divider restyling, (5) friend request UI (AddContactModal restyle + new MemberPopover component), and (6) a design requirements audit against the Appendix checklist.

The frontend is a React + TypeScript SPA served by Vite, with CSS in `apps/web/src/styles.css`. The backend is NestJS with raw PostgreSQL (no ORM). Existing components already have the correct logic -- the work is primarily restyling and adding visual states, with the notable exception of the ghost contact feature which requires a new migration, backend cascade changes, and a new `deleted_usernames` table.

**Primary recommendation:** Organize work into 6 plans matching the 6 CONTEXT.md work items, with the ghost contact backend changes (migration + cascade) executed first since other UI work has no dependencies on it and it carries the highest risk.

### Pattern 1: Inline Validation Badges (Registration)

**What:** Parse the 409 error from `POST /register` and display field-specific badges next to email/username fields.
**When to use:** After submit, when the server returns a 409 with a message like "email is already registered" or "username is already taken".

The current `register()` API function throws an Error with the message from `data.message`. The error message strings are:
- `"email is already registered"` [VERIFIED: auth.service.ts line 69]
- `"username is already taken"` [VERIFIED: auth.service.ts line 75]

The frontend needs to parse these strings in the catch block and set field-level state. The design shows Badge components with `variant="ok"` (green, "available") and `variant="warn"` (amber, "taken -- try alice1").

**Example:**
```typescript
// In RegisterView.tsx handleSubmit catch block
// Source: current error format from auth.service.ts
catch (err) {
  const msg = err instanceof Error ? err.message : "";
  if (msg.includes("email is already registered")) {
    setEmailStatus("taken");
    setUsernameStatus("available"); // if email fails, username wasn't the problem
  } else if (msg.includes("username is already taken")) {
    setUsernameStatus("taken");
    setEmailStatus("available"); // email passed uniqueness check
  } else {
    setError(msg || "Registration failed");
  }
}
```

**Important caveat:** The backend checks email first, then username. If email is taken, the username check never runs. So we can only show "available" for a field if the error specifically names the OTHER field. If email fails, username status is unknown (not necessarily available). The implementation should handle this accurately -- show "taken" badge only for the failing field, and no badge for unchecked fields. [VERIFIED: auth.service.ts lines 66-76 -- sequential checks]

### Anti-Patterns to Avoid
- **Overloading the error string parsing:** Don't try to extract detailed field info from error messages beyond what the backend sends. The 409 response has a simple `message` string, not structured field errors. Parse the known strings, fall back to generic error for anything else.
- **Breaking existing frozen detection:** DmChatView already handles `frozen` and `ineligibleReason` state. The redesign should enhance the UI rendering without changing the state management logic.
- **Deleting DM conversations on account deletion:** The current code deletes DM conversation rows. The new behavior must KEEP them and set `frozen=true` instead. This is a behavior change, not just UI.

## Common Pitfalls

### Pitfall 1: Backend checks email before username -- status inference is limited
**What goes wrong:** Frontend assumes both fields are validated and shows "available" for the non-failing field.
**Why it happens:** `auth.service.register()` checks email first, throws immediately if email is taken, never reaches username check.
**How to avoid:** Only show "taken" for the specific failing field. Don't show "available" for the other field unless the registration succeeded (which means both are available). Show no status badge for unchecked fields.
**Warning signs:** User sees "available" badge on username when email was the failure -- the username was never actually checked.

### Pitfall 2: dm_conversations FK CASCADE deletes conversation rows on user deletion
**What goes wrong:** The `dm_conversations` table has `ON DELETE CASCADE` on both `user_a_id` and `user_b_id` foreign keys. When the user record is deleted, the DM conversation rows are automatically cascaded away by the database, regardless of application-level freezing.
**Why it happens:** Migration 0004 defines `user_a_id UUID NOT NULL REFERENCES users (id) ON DELETE CASCADE`.
**How to avoid:** The migration for ghost contacts MUST `ALTER TABLE dm_conversations` to change the FK action from `CASCADE` to `SET NULL` (and make the columns nullable), or introduce an alternative column strategy. Otherwise, the freeze step is pointless because the final `DELETE FROM users` will cascade-delete all DM conversations anyway.
**Warning signs:** DM conversations disappear after account deletion despite the freeze step.

### Pitfall 3: Popover positioning in scrollable containers
**What goes wrong:** MemberPopover renders off-screen or behind scroll boundaries.
**Why it happens:** Absolute positioning relative to a scrollable parent doesn't account for viewport edges.
**How to avoid:** Use `getBoundingClientRect()` to calculate position relative to viewport. Keep it simple -- position below the clicked row, flip upward if near bottom of viewport.
**Warning signs:** Popover clipped at bottom of members panel.

### Pitfall 4: Divider restyle breaks load-older functionality
**What goes wrong:** The "older messages" divider is currently a `<button>` that triggers `onLoadOlder`. Restyling to a `chat-divider` pattern could lose the click handler.
**Why it happens:** The design shows a visual divider, but the current implementation is an interactive button.
**How to avoid:** Keep the `<button>` element with the click handler, but restyle it to look like the design's `chat-divider` (dashed lines + centered text). The element can be a button styled as a divider.
**Warning signs:** "older messages" no longer clickable after restyle.

### Pitfall 5: Ghost contact -- frozen_reason column needed
**What goes wrong:** DmChatView cannot distinguish between "frozen because banned" and "frozen because account deleted" to show the correct banner.
**Why it happens:** The current `dm_conversations.frozen` is a simple boolean with no reason column.
**How to avoid:** Add a `frozen_reason` column (e.g., `TEXT CHECK (frozen_reason IN ('banned', 'account_deleted'))`) or a `deleted_at` timestamp column to the migration. The frontend needs this to decide between "CONVERSATION FROZEN" and "ACCOUNT DELETED" banners.
**Warning signs:** All frozen DMs show the same banner regardless of reason.

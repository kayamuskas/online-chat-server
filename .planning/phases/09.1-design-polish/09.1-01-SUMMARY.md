# Phase 9.1 Plan 01: Ghost Contact Backend Summary

**One-liner:** DB migration adds `deleted_usernames` table, `frozen_reason`/`frozen_at` columns, and FK SET NULL on `dm_conversations`; `deleteAccount` cascade now freezes DMs and reserves usernames; registration rejects previously-used usernames.

## What Was Built

### Migration `0009_ghost_contact.sql`
- Creates `deleted_usernames` table with `username UNIQUE` constraint
- Adds `frozen_reason TEXT CHECK (IN ('banned', 'account_deleted'))` and `frozen_at TIMESTAMPTZ` to `dm_conversations`
- Changes `dm_conversations.user_a_id` and `user_b_id` FKs from `ON DELETE CASCADE` to `ON DELETE SET NULL` (makes columns nullable first)
- Backfills `frozen_reason = 'banned'` for pre-existing frozen conversations

### `ContactsRepository.freezeDmConversationsFor(userId)`
Freezes all unfrozen DM conversations where the user is a participant: sets `frozen=TRUE`, `frozen_reason='account_deleted'`, `frozen_at=NOW()`.

### `UserRepository.isUsernameReserved(username)` / `reserveUsername(username)`
Checks and populates the `deleted_usernames` table to block re-registration of previously-used usernames.

### `AuthService.deleteAccount` changes
1. Replaced `deleteDmConversationsFor` call with `freezeDmConversationsFor` (D-9.1-14)
2. Added `reserveUsername(user.username)` before `deleteById` (D-9.1-13)

### `AuthService.register` changes
Added `isUsernameReserved` check after active-username uniqueness check, before `hashPassword` (D-9.1-13).

### `DmConversation` interface (api.ts)
Updated to include `frozen_reason: 'banned' | 'account_deleted' | null`, `frozen_at: string | null`, and nullable `user_a_id`/`user_b_id` (both become `string | null`).

## Deviations from Plan

None — plan executed exactly as written.

## Threat Surface Scan

No new network endpoints introduced. The `deleted_usernames` table is an internal server-side table only accessible through the `UserRepository` methods. The FK SET NULL change is schema-only. No new trust boundary surfaces.

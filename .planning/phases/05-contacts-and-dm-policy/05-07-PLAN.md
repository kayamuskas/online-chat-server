<objective>
Close the Phase 5 UAT gap where a blocked user can still appear in incoming friend requests, causing the same person to show up in both `Blocked Users` and `Incoming Requests`.

Purpose: Phase 5 already enforces bans for DM eligibility, but the friend-request lifecycle is still partially ban-blind. This plan makes ban policy consistent across request creation and request visibility so blocked relationships cannot continue to behave like pending social connections.
Output: one focused backend consistency fix plus regression tests and a human recheck of the exact blocked-user scenario reported in UAT.
</objective>

<interfaces>
Existing implementation to preserve and extend:

- `ContactsService.checkDmEligibility()` already treats any ban in either direction as disqualifying for DM access.
- `ContactsService.sendFriendRequest()` currently checks self-request, friendship, and duplicate-request conditions, but does not reject ban-protected relationships.
- `ContactsRepository.listIncomingRequests()` currently returns pending requests by `target_id` without suppressing rows whose requester is already blocked by the recipient (or vice versa).
- The frontend request surfaces (`FriendRequestDropdown`, `ContactsView`) already render whatever the API returns; this gap should be solved primarily at the API/domain layer rather than with client-side hiding.

This gap-closure plan must not add new routes, new tables, or Phase 6 messaging behavior. Reuse the existing contacts endpoints and ban model.
</interfaces>
